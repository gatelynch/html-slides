# 草案 outline 格式

生成整份 HTML 前，先產出一份草案 markdown，逐頁簡述重點，交給使用者確認。草案的目的是讓「頁序、版型、每頁要講什麼」在便宜的階段就能討論與修改，不必等 HTML 做完才發現方向不對。

## 原則

- 草案是**討論用的骨架**，不是完整文案；每頁重點用短句，不要先把內文寫滿。
- 一頁一個區塊，標出版型（用 `deck-patterns.md` / `teaching-patterns.md` 的 22 種頁型名稱）。
- 相鄰頁避免連用同一種版型；教學簡報要看得出「嘗試 → 揭示 → 比較 → 修正」的節奏。
- 需要圖片、截圖或資料時，在該頁標出來，提醒使用者準備。
- 檔名用 `<主題>-outline.md`，放在簡報預定輸出的相鄰位置。

## 結構

開頭放一段 meta，之後每頁一個 `##` 區塊。

```markdown
# <簡報標題> — 草案

- 目的：<為什麼做這份簡報、希望觀眾改變什麼>
- 受眾：<對象與程度>
- 風格：Clean
- 規模：約 12 頁 / 約 20 分鐘
- 講者備註：需要

---

## 1. Cover — 把工具用成可教的流程
- 重點：一句承諾；今天會帶走一套可重複的方法
- 備註：開場用一句話說明今天會完成什麼
- 資產：無

## 2. Agenda — 今天的路徑
- 重點：看見問題 / 做一次任務 / 帶走流程

## 3. Stat — 三個數字讓重點站穩
- 重點：4 分鐘嘗試、3 個判準、1 個產出
- 資產：若有真實數據，補上來源

## 4. Activity — 請幫導師回覆一則訊息
- 情境：家長對換座位不滿，傳來情緒強烈的訊息
- 任務：寫一版 120 字以內回覆，兼顧同理、事實、下一步
- 時間：個人 4 分鐘 + 小組比較 3 分鐘
- 產出：一段可直接傳出的訊息
- 討論：哪一句最能降溫？哪一句可能讓家長覺得被敷衍？

## 5. Reveal — 教師版本為什麼比較穩
- 重點：先接住情緒 / 只說已知事實 / 給可追蹤的下一步
- 備註：接在學生嘗試之後再揭示

## 12. Closing — 判斷仍在你身上
- 重點：工具是幫手；把每次使用設計成可觀察、可比較、可修正的活動
- 資產：QR code 或連結（可選）
```

## 每頁可用的欄位

- **重點**：一般頁用 2–5 條短句，描述這頁要傳達什麼。
- **情境 / 任務 / 時間 / 產出 / 討論**：教學活動頁（Activity、Try First 等）改用這組欄位，對應 `teaching-patterns.md` 的共同要求。
- **備註**：這頁的講者提示（若簡報需要備註）。
- **資產**：需要的圖片、截圖、數據與來源；沒有就寫「無」或省略。

## 確認後

使用者確認草案後，再依 `references/style-system.md` 與 `assets/templates/base-deck.html` 把每個區塊轉成 `<section class="slide">`。草案裡標的版型、標題與重點，就是生成 HTML 時的對照表。
