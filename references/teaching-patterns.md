# 教學活動簡報設計

教學簡報不是把重點講完，而是安排學習者如何進入任務、嘗試、卡住、看見範例、修正想法。

## 教學頁型的共同要求

每個教學活動至少要交代：

- **情境**：學習者現在扮演誰？面對什麼問題？
- **任務**：要做什麼，不要只寫「討論一下」。
- **限制**：時間、字數、工具、角色或不可做的事。
- **產出**：最後要留下什麼，例如一句回覆、一段 prompt、一張表。
- **討論問題**：做完後要比較什麼、反思什麼。

## 10 種教學版型

### Activity

完整任務頁。適合現場活動、分組討論與實作。

- 結構：情境、任務、時間、產出、討論問題。
- 規格：任務句要可執行；產出要可收集或可分享。
- 常見錯誤：只有概念，沒有要求學習者做出東西。

### Try First

先嘗試頁。適合訓練判斷力與創作能力。

- 結構：輸入材料、限制、作答形式。
- 規格：不要先給完整範本；只給必要背景。
- 常見錯誤：題目太大，學習者不知道第一步。

### Reveal

揭示頁。接在 Try First 後面，用來比較學生版本與教師版本。

- 結構：範例、有效原因、可改進處。
- 規格：不要只公布答案，要指出判斷依據。
- 常見錯誤：揭示太早，讓學生失去嘗試機會。

### Prompt Lab

Prompt 或 AI 工具實驗頁。

- 結構：目標、prompt、變因、觀察點。
- 規格：一次只改一個變因，例如角色、限制、輸出格式或評分標準。
- 常見錯誤：prompt 很長但沒有觀察任務。

### Compare Tools

工具或方法比較頁。

- 結構：共同任務、比較維度、建議使用情境。
- 規格：比較維度要一致，例如速度、品質、可控性、成本、隱私。
- 常見錯誤：用喜好代替判準。

### Rubric

品質規準頁。

- 結構：評估維度、等級、可觀察行為。
- 規格：每個等級都要有具體描述，而不是只寫好 / 普通 / 差。
- 常見錯誤：規準太抽象，無法用來給回饋。

### Classroom Case

課堂案例頁。

- 結構：角色、情境、衝突、決策點。
- 規格：案例要像真實現場，讓學習者能代入。
- 常見錯誤：案例太乾淨，沒有需要判斷的張力。

### Reflection

反思頁。

- 結構：個人問題、同儕分享、下一步。
- 規格：問題要能引出行動，例如「下次我會先改哪一步？」
- 常見錯誤：只問心得，沒有連回任務。

### Workflow

可重複操作的流程頁。

- 結構：輸入、步驟、輸出、檢查點。
- 規格：每一步都要有可觀察結果。
- 常見錯誤：流程太抽象，缺少檢查點。

### Before/After

前後對照頁。

- 結構：修改前、修改後、改動理由。
- 規格：標出 2 到 4 個關鍵差異，不要只讓觀眾自己看。
- 常見錯誤：前後都很長，觀眾看不出差別。

## 建議教學節奏

如果活動目標是訓練判斷力，建議節奏是：

1. 給情境與任務。
2. 讓學生自己嘗試。
3. 讓學生互相比較。
4. 再揭示範例或教師版本。
5. 討論範例為什麼有效，以及還能怎麼改。

HTML 簡報可以把「學生嘗試頁」與「範例揭示頁」拆成兩頁，而不是把所有內容塞在同一頁。

## 延續同一情境

若同一個情境會延伸成多個活動，應讓每個活動像階梯：

- 活動一：先處理最表層任務。
- 活動二：加入限制、判準或角色差異。
- 活動三：要求學習者修改、評估或重設計。

例如「五年級導師回覆生氣家長」可以延伸成：

- 先寫一版回覆。
- 再請 AI 協助調整語氣與資訊完整度。
- 最後評估回覆是否兼顧同理、事實、後續處理與教師專業。

## 避免把活動做成講述

以下頁面通常需要重做：

- 只有三個重點，沒有任務。
- 只有 prompt 範本，沒有讓學生先嘗試。
- 只有概念說明，沒有產出物。
- 只有教師答案，沒有比較與修正。
